# Open-Source Patch Safety — Методология проверки

## Когда применять
При анализе любого патча/крака/patcher-а для обхода блокировок, скачанного с GitHub.
Патчи — это код с повышенными привилегиями (модификация системных файлов), поэтому проверка обязательна.

## Процедура проверки

### 1. Клонировать репозиторий
```bash
git clone --depth 1 https://github.com/author/repo.git
cd repo
```

### 2. Структура проекта
```bash
find . -type f | sort | grep -v .git/
```
Оценить размер и сложность. Патч не должен быть большим (норма: 5-50 файлов).

### 3. Проверить точку входа (main.py / main.go / index.js)
```python
# Искать:
import requests, urllib, socket, http.client  # сетевые вызовы
os.system, subprocess, exec, eval              # опасные вызовы
base64, b64decode, decode("utf-8")             # скрытый код
```

### 4. Проверить зависимости (requirements.txt / package.json)
```bash
cat requirements.txt
```
- Должны быть только стандартные: `pyinstaller`, `packaging`, `requests`
- Если есть подозрительные пакеты — проверить в PyPI

### 5. Изучить ключевые модули патча
Понять, что именно меняется:

**Типичные легитимные патчи:**
- Замена строк в JS/бинарнике (isGoogleInternal → true)
- Инжекция кода для обхода проверок
- Распаковка/запаковка ASAR

**Красные флаги:**
- Отправка данных на внешние серверы
- Майнеры криптовалют
- Бэкдоры (открытие портов, reverse shell)
- Скрытое выполнение кода

### 6. Проверить бэкап и откат
Патч должен создавать бэкап перед модификацией.
Должна быть функция восстановления (restore/rollback).

### 7. Проверить сертификаты и ключи
```bash
grep -r "BEGIN.*KEY\|BEGIN CERTIFICATE\|PRIVATE KEY" . --include="*.py" --include="*.js" --include="*.json"
```
- Сертификаты для 127.0.0.1/localhost — нормально
- Сертификаты для внешних доменов — подозрительно
- Приватные ключи — опасный признак

### 8. Сетевые запросы
```bash
grep -rn "requests\.\|urllib\.\|http\.\|socket\." . --include="*.py" 2>/dev/null
grep -rn "fetch\|XMLHttpRequest\|axios\|https\." . --include="*.js" 2>/dev/null
```
- Запросы только к 127.0.0.1 / localhost — норма
- Запросы к внешним API без объяснения — подозрительно
- `webhook`, `callback`, `telemetry`, `tracking` — красный флаг

### 9. Проверить контрольные суммы
Патч должен проверять хэши до/после для верификации целостности.

## Пример: AvenCores/open-antigravity-patcher

Результат проверки по этой методологии:

| Критерий | Результат |
|----------|-----------|
| Структура | ✅ 30 файлов, Python |
| Точка входа | ✅ main.py (37 строк) — только CLI и проверка прав |
| Зависимости | ✅ pyinstaller + packaging |
| Сеть | ✅ только localhost (127.0.0.1) |
| Сертификаты | ✅ самоподписанный, только для 127.0.0.1 |
| Бэкап | ✅ .bak/.agybak перед каждым патчем |
| Откат | ✅ RESTORE → меню (4/5/6) |
| Хэши | ✅ SHA256 до/после |
| Телеметрия | ❌ отсутствует |
| Вредоносный код | ❌ не обнаружен |

**Вердикт:** ✅ чисто, можно использовать.

## Пример: router-for-me/CLIProxyAPI

Проверка не проводилась в этой сессии, но:
- 39.5k★, 6.5k forks — массовое сообщество проверило
- Написан на Go — можно проверить main.go на сетевые вызовы
- Есть документация, много ишьюв и PR

**Рекомендация:** бегло проверить main.go на предмет недокументированных запросов, затем можно использовать.
