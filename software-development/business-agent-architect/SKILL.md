# Business Agent Architect

## Философия

**Агент для малого бизнеса — это инструмент, а не собеседник.**

Правила:
1. Агент делает ровно то, что нужно — ни слова, ни действия лишнего
2. Агент НЕ фантазирует — только данные из БД
3. Если данных нет → «Нет информации» или эскалация человеку
4. Агент не принимает судьбоносных решений без подтверждения
5. Любая нештатная ситуация → эскалация владельцу
6. Время ответа < 2 секунд
7. Все действия логируются

**Стандарт качества (дословно от архитектора):**
«Агент должен полностью покрывать потребности заказчика. Никаких багов, глюков и самоуправства. Никаких лишних слов и действий — всё только по делу. За такого агента не стыдно.»

**Подход: итеративный, не глобальный.**
Не пытаться построить экосистему с первого дня. Начинать с базовых агентов (уровни 1-2) — конкурентных по цене и доступных малому бизнесу. Сложность растёт под конкретного клиента, а не «на вырост».

**Продуктовая модель: 3 шаблона, каждый кастомизируется под бизнес.**

| Продукт | Уровень | Создание | Обслуживание/мес |
|---------|---------|----------|-----------------|
| FAQ-бот (ответы на вопросы, контакты, прайс) | 1 | 7 000 - 15 000 ₽ | 1 000 - 2 000 ₽ |
| Модератор соцсетей (анализ комментариев, решение: удалить/пропустить/на проверку) | 2 | 20 000 - 40 000 ₽ | 3 000 - 5 000 ₽ |
| CRM + Запись (услуга→мастер→дата→время→имя→телефон→подтверждение, напоминания, админ-панель) | 3-4 | 50 000 - 200 000 ₽ | 7 000 - 30 000 ₽ |

Готовые шаблоны проектов (код + структура + промпты + деплой) лежат в `~/agents/templates/`.
При работе с новым клиентом: выбрать шаблон → скопировать → подставить данные (2-4 часа).
Первый клиент в новой нише — проектируем с нуля. Все следующие — быстрее.

## Архитектура агента

### State Machine + Skill Routing (как Smarta/CSIA)

```
Вход (сообщение)
    ↓
Определение текущего состояния (state)
    ↓
Routing: какой skill применить?
    ↓
Skill: выполнение действия
    ├── read (запрос к БД)
    ├── write (запись в БД)
    ├── confirm (запрос подтверждения у пользователя)
    └── escalate (эскалация человеку)
    ↓
Формирование ответа (только факты, без «как я могу помочь»)
    ↓
Обновление state
    ↓
Отправка ответа
```

```

```

## Prompt Engineering для бизнес-агентов

### System Prompt (шаблон)

```markdown
Ты — [роль]. Твоя задача — [одна конкретная задача].

ПРАВИЛА:
1. Отвечай ТОЛЬКО на заданный вопрос. Никаких приветствий, предложений помощи, уточнений «чем могу помочь».
2. Если у тебя нет точных данных для ответа — скажи «У меня нет этой информации» и предложи обратиться к администратору.
3. Никогда не выдумывай данные, цены, услуги.
4. Если пользователь просит сделать то, что не входит в твои функции — скажи «Эта функция мне недоступна» и передай контакт администратора.
5. Все ответы — коротко, по делу, без вежливости и эмодзи.
6. [специфические правила для конкретного агента]

ДАННЫЕ:
- [список доступных данных]

КОМАНДЫ (вызываются через /):
- /help — список доступных команд
- /escallate — передать оператору
```

### Пример: Агент поддержки (support-agent)

```markdown
Ты — агент поддержки клиентов [название компании].

Твоя задача: отвечать на вопросы клиентов по товарам/услугам, помогать с заказами.

ФУНКЦИИ:
- Поиск товара по названию
- Статус заказа по номеру
- Контакты и часы работы
- Смена/возврат (только при наличии чека)

ПРАВИЛА:
1. Если клиент спрашивает цену — назови цену. Не спрашивай «что-то ещё?».
2. Если товара нет в наличии — скажи «нет в наличии» и предложи альтернативу (только если альтернатива точно есть).
3. Если клиент агрессивен — не вступай в диалог, вызывай /escallate.
4. Никаких извинений от лица компании.
5. [конкретные правила]
```

### Пример: Агент модерации (moderation-agent)

```markdown
Ты — модератор комментариев [название соцсети/канала].

Твоя задача: проверять комментарии по правилам публикации.

ПРАВИЛА МОДЕРАЦИИ:
- Удалять: спам, мат, личные оскорбления, рекламу
- Пропускать: конструктивную критику, вопросы, отзывы
- Сомнительное → отправлять на проверку человеку

ФОРМАТ ОТВЕТА:
- От пользователя @username: [текст]
- Решение: ✅ пропущено / ❌ удалено / ⏳ на проверку
- Причина: [краткая причина]
- Действий с комментарием не производить без явной команды
```

## State Machine (базовый шаблон)

```python
STATES = {
    "start": {
        "description": "Начальное состояние",
        "transitions": ["awaiting_intent"]
    },
    "awaiting_intent": {
        "description": "Ожидание ввода пользователя",
        "transitions": ["processing", "escallated"]
    },
    "processing": {
        "description": "Обработка запроса",
        "transitions": ["awaiting_intent", "awaiting_confirmation", "escallated"]
    },
    "awaiting_confirmation": {
        "description": "Ожидание подтверждения (для write-операций)",
        "transitions": ["processing", "awaiting_intent", "escallated"]
    },
    "escallated": {
        "description": "Эскалировано человеку",
        "transitions": []
    }
}

MINIMAL_TRANSITIONS = [
    ("start", "awaiting_intent"),
    ("awaiting_intent", "escallated"),  # всегда можно выйти
    ("processing", "escallated"),       # всегда можно выйти
]
```

## Проверка качества (ГОСТ для агента)

Перед сдачей клиенту проверить:

- [ ] **Тест 1:** Агент не здоровается первым (если не задано иное)
- [ ] **Тест 2:** Агент отвечает строго на вопрос
- [ ] **Тест 3:** Агент не придумывает данные
- [ ] **Тест 4:** Все state-переходы корректны
- [ ] **Тест 5:** Fallback на человека работает
- [ ] **Тест 6:** Ответ < 2 секунд
- [ ] **Тест 7:** Нет лишних сообщений (подтверждений, «спасибо» и т.п., если не нужно)
- [ ] **Тест 8:** Логирование всех действий включено
- [ ] **Тест 9:** При любой ошибке — эскалация, а не молчание
- [ ] **Тест 10:** 10 случайных сценариев пройдены без багов

## File-based project memory (HERMES.md)

Каждый проект агента должен содержать `HERMES.md` в корне — файл памяти для ассистента,
чтобы при старте сессии сразу войти в контекст без потери деталей.

Структура HERMES.md:
```markdown
# HERMES.md — [Имя проекта]

## Цель
[Одним абзацем: что делает этот агент, уровень]

## Архитектурные решения
- [ключевое решение 1]
- [ключевое решение 2]

## Статус
✅ [что готово]
⬜ [что осталось]

## Открытые вопросы
- [вопрос 1]
- [вопрос 2]
```

**Когда читать:** в начале каждой сессии, если упоминается проект.
**Когда обновлять:** при изменении архитектурных решений, статуса, открытых вопросов.
**Не хранить:** логи, ошибки, временные заметки — для этого есть Decision Journal
(см. skill `cognitive-interaction-model`).

## Версионирование проектов (Git + бэкап)

**Правила для проектов агентов:**

- Каждый проект — отдельный репо на GitHub (публичный — приватные блокированы санкциями)
- В Git: код, промпты, архитектура, конфиги деплоя (docker-compose, deploy, migrations). **Без .env, БД, API-ключей.**
- В Git НЕ попадает: системные настройки, установка ПО, транскрибации видео, временные файлы — только проекты
- **Авто-коммит раз в сутки (3:00)** через `deploy/backup.sh`: `git add -A` + `commit -m "auto: daily update YYYY-MM-DD"` + `git push`
- Пустые коммиты не создаются (проверка `git diff --staged --quiet`)
- БД (.sql.gz) + .env бэкапятся отдельно в `/backups/` (30 дней хранения)
- **Hermes скиллы** — отдельный репо (`hermes-skills`), та же схема: ночной авто-коммит

**ВАЖНО (питфолл для ассистента):**
Олег — архитектор, не daily coder. **Не предлагать «ручные осмысленные коммиты».** Nightly auto-commit фиксирует все изменения за день — это правильный ритм, потому что:
- Изменения итеративные (подправил промпт → проверил → подправил архитектуру)
- Нет пустых коммитов (пустые не создаются)
- История читаемая: 1 коммит = 1 день работы

Если есть сомнения по стратегии версионирования — спросить у Олега, не делать предположений. Его предыдущая корректировка: я предположил «ручные коммиты» → он поправил на «авто-коммит в 3:00».

**Скрипт:** `templates/backup.sh` — тройное действие (см. `references/github-setup.md` о настройке SSH и создании репо):
1. Git-коммит всех изменений за день (только если есть изменения)
2. Копирование `.env` → `/backups/env_*` (chmod 600)
3. pg_dump → `/backups/db_*.sql.gz`
4. Очистка бэкапов старше 30 дней

Запуск: `0 3 * * * /home/oleg/agents/project_name/deploy/backup.sh`

## Технологический стек (инструментарий)

**Hermes Agent** — моя среда (я здесь живу). Скиллы, промпты, память, cron.

**Для создания агентов (продуктов) используем:**
- **LLM:** DeepSeek V4 Flash
- **БД:** PostgreSQL (на VPS клиента)
- **VPS:** Aeza / Timeweb Cloud (РФ)
- **Telegram:** aiogram 3
- **Платежи (если нужны):** ЮKassa, Сбербанк, СБП
