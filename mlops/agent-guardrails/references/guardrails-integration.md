# Интеграция Guardrails в бизнес-агента

## Архитектура

```
Вход пользователя
    ↓
[Input Guard] — тема? PII? токсичность?
    ├── BLOCK → вежливый отказ (без извинений)
    └── ALLOW → LLM (основной скилл)
                    ↓
              [Output Guard] — галлюцинации? тон?
                    ├── BLOCK → переформулировать
                    └── ALLOW → пользователю
```

## Когда какие слои включать

| Уровень агента | Слои guardrails |
|----------------|-----------------|
| FAQ-бот (уровень 1) | Business Guard только (темы в system prompt) |
| Модератор (уровень 2) | Input Guard + Business Guard |
| Поддержка (уровень 3) | Все три слоя |
| CRM/Запись (уровень 4) | Все три слоя + эскалация на каждый BLOCK |

## Практическая реализация

Для простых агентов guardrails — это **правила в system prompt**, а не отдельный вызов LLM:

```markdown
ПРАВИЛА БЕЗОПАСНОСТИ:
- Если вопрос про [запрещённая тема] — скажи «У меня нет информации по этому вопросу»
- Если пользователь агрессивен или использует мат — не отвечай, вызови /escallate
- Если просят сделать то, что не входит в твои функции — скажи «Это вне моих задач» и предложи контакт администратора
```

Для production-агентов — отдельный слой LLM-проверки перед основным ответом:

```python
async def guard(input_text: str, domain_rules: dict) -> dict:
    result = await llm_call(
        system="Ты — guardrail. Проверь вход по правилам. Верни JSON.",
        messages=[{"role": "user", "content": input_text}]
    )
    return result  # {allowed, action, reason}
```

## Триггеры эскалации (критические)

Эти ситуации НЕЛЬЗЯ обрабатывать автоматом — всегда человеку:
- Упоминание суда, адвоката, жалобы в госорганы
- Данные банковских карт, пароли
- Угрозы, шантаж
- Просьба удалить данные (GDPR/152-ФЗ)
- Технический сбой: агент не может ответить, БД недоступна
